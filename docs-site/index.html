VNE-CLI

Generate complete VoidNovelEngine visual novel projects from novel text using AI.

Python 3.10+ MIT License pip install vne-cli

Quick Start

Transform a novel into a playable visual novel in three commands:

# Install
pip install vne-cli

# 1. Extract story structure (characters, scenes, dialogue, choices)
vne extract novel.txt -o story.json

# 2. Generate character sprites and scene backgrounds
vne generate-assets story.json -o ./assets/

# 3. Assemble into a VNE project
vne assemble story.json --assets ./assets/ -o ./my-vn-project/

The output at ./my-vn-project/ is a complete VoidNovelEngine project you can open in the VNE editor or run directly.

Dry Run

Preview what each stage will do without making API calls:

vne extract novel.txt --dry-run
vne generate-assets story.json --dry-run

For AI Agents

If you are an AI assistant helping a user generate a visual novel, here is exactly what to do.

Follow the steps below in order. Each step includes the exact command to run and what to expect.

Prerequisites

  1. Ensure Python 3.10+ is installed: python3 --version
  2. Install VNE-CLI: pip install vne-cli
  3. Verify installation: vne --version
  4. The user needs an LLM API key (OpenAI or Anthropic) and an image generation API key (OpenAI/DALL-E or Stability AI)

Step 1: Configure Providers

Initialize the global config:

vne config init --global

This creates ~/.vne-cli/config.toml. Edit it to set up providers. Here is a recommended configuration using OpenAI for both LLM and image generation:

# ~/.vne-cli/config.toml

[providers.llm]
package = "vne_cli.providers.llm.openai_provider"
factory = "create_llm_provider"
model = "gpt-4o"

[providers.image]
package = "vne_cli.providers.image.dalle_provider"
factory = "create_image_provider"
model = "dall-e-3"

Set the API key as an environment variable (do not put secrets in config files if possible):

export VNE_CLI_OPENAI_API_KEY="sk-..."

Alternative: use Anthropic for the LLM:

[providers.llm]
package = "vne_cli.providers.llm.anthropic_provider"
factory = "create_llm_provider"
model = "claude-sonnet-4-20250514"
export VNE_CLI_ANTHROPIC_API_KEY="sk-ant-..."

Verify the configuration:

vne config show --resolved

Step 2: Prepare the Input

The user provides a novel as a .txt or .md file. If the novel has chapters, markdown headings (#) will be detected as chapter boundaries. Explicit branching choices can be marked up:

[CHOICE: What should Elena do?]
[OPTION: Read the letter immediately]
[OPTION: Hide it and read later]

The parser also detects implicit branching cues from narrative patterns.

Step 3: Extract Story Structure

vne extract novel.txt -o story.json --verbose

What this produces: A story.json file containing extracted characters (with expression variants), scenes, dialogue, narration, branching choices, and cinematic annotations.

Expected duration: Depends on novel length. A 50,000-word novel takes roughly 2-5 minutes with GPT-4o.

Optional: extract only characters first for review:

vne extract novel.txt --characters-only -o characters.json

Step 4: Generate Assets

vne generate-assets story.json -o ./assets/ --verbose

What this produces:

  • assets/characters/*.png -- Character sprites for each expression variant
  • assets/backgrounds/*.png -- Scene background images
  • assets/asset-manifest.json -- Progress tracking manifest

Expected duration: Depends on the number of unique characters and scenes. Typically 5-20 minutes. DALL-E 3 generates one image at a time; Stable Diffusion supports concurrency.

Resume after failure: If the process crashes or is interrupted, re-run the same command. It reads the manifest and skips already-generated assets.

Options for controlling generation:

# Only generate character sprites
vne generate-assets story.json -o ./assets/ --characters-only

# Only generate backgrounds
vne generate-assets story.json -o ./assets/ --backgrounds-only

# Override art style
vne generate-assets story.json -o ./assets/ --style "watercolor"

# Increase concurrency (default: 3)
vne generate-assets story.json -o ./assets/ --concurrency 5

Step 5: Assemble the Project

vne assemble story.json --assets ./assets/ -o ./my-vn-project/ --verbose

What this produces: A complete VoidNovelEngine project directory:

my-vn-project/
  project.vne              # Engine manifest (JSON)
  main.lua                 # Engine entry point
  application/
    flow/
      entry.flow           # Entry flow with fade-in
      ch_001_sc_001.flow   # Scene flow files
      ...
    resources/
      characters/          # Character sprite PNGs
      backgrounds/         # Background PNGs
      audio/               # Audio files (if any)
      fonts/               # Font files
    icon/
      icon.png             # Default 64x64 icon

Assembly options:

# Set game window resolution
vne assemble story.json --assets ./assets/ -o ./project/ --resolution 1920x1080

# Set project title
vne assemble story.json --assets ./assets/ -o ./project/ --title "The Crystal Kingdom"

# Disable cinematic effects
vne assemble story.json --assets ./assets/ -o ./project/ --no-cinematic

# Use minimal cinematic tier
vne assemble story.json --assets ./assets/ -o ./project/ --cinematic-tier base

Step 6: Deliver the Result

Tell the user their project is at the output directory. They can:

  • Open it in the VoidNovelEngine editor
  • Run it directly with the VNE runtime
  • Edit .flow files manually for fine-tuning (see the Flow Format section)

Complete Example: End-to-End

# Setup (one-time)
pip install vne-cli
vne config init --global
# Edit ~/.vne-cli/config.toml to set providers (see Step 1)
export VNE_CLI_OPENAI_API_KEY="sk-..."

# Pipeline
vne extract novel.txt -o story.json
vne generate-assets story.json -o ./assets/
vne assemble story.json --assets ./assets/ -o ./my-vn-project/

# Done! Project is at ./my-vn-project/

Troubleshooting for AI Agents

ProblemSolution
API key errorCheck VNE_CLI_OPENAI_API_KEY (or the relevant provider key) is set in the environment
Rate limitingVNE-CLI has built-in retry with exponential backoff (3 retries, max 30s delay). If persistent, reduce --concurrency
Asset generation interruptedRe-run the same vne generate-assets command. It resumes from the manifest automatically
Missing characters in outputRun vne extract --characters-only first to review extracted characters, then pass with --characters characters.json
Config not loadingRun vne config show --resolved to see which config layer each value comes from

Command Reference

vne extract

Parse novel text into structured story JSON.

vne extract <INPUT_FILE> [OPTIONS]
Argument / OptionDescription
INPUT_FILEPath to novel text file (.txt, .md)
-o, --output PATHOutput story.json path (default: ./story.json)
--characters-onlyRun only the character extraction pre-pass
--characters PATHReuse an existing character registry JSON
--config PATHProject config file override
--max-chapters INTOverride max chapter count
--max-branch-depth INTOverride max branch depth
--dry-runShow extraction plan without calling the LLM
--verbose / --no-verboseEnable detailed logging

What it does:

  1. Reads the input file and detects chapter boundaries
  2. Chunks text to fit within the LLM context window (respecting chapter boundaries)
  3. Runs a character extraction pre-pass across all chunks, then merges and deduplicates
  4. Extracts chapters, scenes, dialogue, narration, and cinematic annotations
  5. Detects branching cues (explicit [CHOICE: ...] markers and implicit narrative patterns)
  6. Enforces branch depth and choice count limits
  7. Validates the output (character references, branch targets, orphaned scenes)
  8. Writes story.json

vne generate-assets

Generate images for characters and backgrounds via a pluggable image generation API.

vne generate-assets <STORY_JSON> [OPTIONS]
Argument / OptionDescription
STORY_JSONPath to story.json from the extract stage
-o, --output PATHOutput assets directory (default: ./assets/)
--manifest PATHResume from an existing asset manifest
--characters-onlyGenerate only character sprites
--backgrounds-onlyGenerate only backgrounds
--style TEXTOverride image style (e.g., "watercolor")
--concurrency INTMax parallel API requests (default: 3)
--config PATHProject config file override
--dry-runShow what would be generated without calling APIs
--verbose / --no-verboseEnable detailed logging

What it does:

  1. Reads story.json and builds asset requests (one per character expression + one per unique background)
  2. Deduplicates backgrounds by location and time-of-day
  3. Initializes or loads an asset manifest for resume tracking
  4. Generates images concurrently with a configurable semaphore bound
  5. Saves the manifest after each asset for crash recovery
  6. Outputs to assets/characters/ and assets/backgrounds/

vne assemble

Assemble story JSON and assets into a ready-to-run VNE project.

vne assemble <STORY_JSON> --assets <DIR> [OPTIONS]
Argument / OptionDescription
STORY_JSONPath to story.json from the extract stage
--assets PATHAssets directory from the generate-assets stage (required)
-o, --output PATHOutput project directory (default: ./project/)
--resolution WxHGame window resolution (e.g., 1920x1080)
--title TEXTOverride project title
--cinematic / --no-cinematicApply cinematic direction layer (default: on)
--cinematic-tier TEXT"base" or "full" cinematic polish
--validate / --no-validateValidate output before writing (default: on)
--config PATHProject config file override
--verbose / --no-verboseEnable detailed logging

What it does:

  1. Generates .flow files for every scene using the flow graph compiler
  2. Creates an entry flow with fade-in and transition to the first scene
  3. Applies cinematic direction (transitions, letterboxing, timing) if enabled
  4. Copies and organizes assets into VNE directory layout
  5. Generates project.vne manifest and main.lua entry point
  6. Validates the assembled project (flow integrity, asset references, entry point)

vne config

Manage VNE-CLI configuration.

vne config show              # Display resolved config as TOML
vne config show --resolved   # Show with source annotations per key
vne config init              # Create ./vne-cli.toml with defaults
vne config init --global     # Create ~/.vne-cli/config.toml

vne validate

Validate any VNE-CLI intermediate file.

vne validate story.json         # Validate extracted story
vne validate scene.flow         # Validate a flow file
vne validate project.vne        # Validate project manifest

vne --version

Display the installed version of VNE-CLI.

export VNE_CLI_OPENAI_API_KEY="sk-..."

# For Anthropic
export VNE_CLI_ANTHROPIC_API_KEY="sk-ant-..."

# For Stability AI

[credentials]
openai_api_key = "sk-..."

Security note: Credentials are never read from project-level config (./vne-cli.toml). This is enforced by the config loader. Prefer environment variables over config files for API keys.

Custom Providers

VNE-CLI uses Python Protocol classes for structural subtyping. Your provider package does not need to import from vne-cli -- it only needs to match the interface.

How the Plugin System Works

  1. You specify a provider in your config TOML (package name + factory function name)
  2. VNE-CLI calls importlib.import_module(package) to load your package
  3. It calls the factory function with any extra config keys as keyword arguments
  4. It verifies the returned object satisfies the protocol via isinstance() (enabled by @runtime_checkable)

LLMProvider Protocol

@runtime_checkable
class LLMProvider(Protocol):
    @property
    def name(self) -> str: ...

    async def complete(
        self,
        prompt: str,
        *,
        system: str | None = None,
        temperature: float = 0.7,
        max_tokens: int = 4096,
        response_format: dict[str, Any] | None = None,
    ) -> str: ...

    async def complete_structured(
        self,
        prompt: str,
        schema: type,
        *,
        system: str | None = None,
        temperature: float = 0.3,
    ) -> Any: ...

    async def close(self) -> None: ...

ImageProvider Protocol

@runtime_checkable
class ImageProvider(Protocol):
    @property
    def name(self) -> str: ...

    async def generate(
        self,
        prompt: str,
        *,
        width: int = 1024,
        height: int = 1024,
        style: str | None = None,
        negative_prompt: str | None = None,
    ) -> bytes: ...

    async def close(self) -> None: ...

Custom LLM Provider Example

# my_llm_provider.py
import json
import httpx

class MyLLMProvider:
    def __init__(self, api_key: str, model: str, base_url: str = "https://api.example.com"):
        self._api_key = api_key
        self._model = model
        self._client = httpx.AsyncClient(
            base_url=base_url,
            headers={"Authorization": f"Bearer {api_key}"},
            timeout=120.0,
        )

    @property
    def name(self) -> str:
        return f"my-provider/{self._model}"

    async def complete(self, prompt, *, system=None,
                       temperature=0.7, max_tokens=4096,
                       response_format=None) -> str:
        payload = {"model": self._model, "prompt": prompt,
                   "temperature": temperature, "max_tokens": max_tokens}
        if system:
            payload["system"] = system
        response = await self._client.post("/v1/completions", json=payload)
        response.raise_for_status()
        return response.json()["text"]

    async def complete_structured(self, prompt, schema, *,
                                  system=None, temperature=0.3):
        schema_json = json.dumps(schema.model_json_schema(), indent=2)
        full_system = (system or "") + f"\n\nRespond with JSON matching:\n{schema_json}"
        raw = await self.complete(prompt, system=full_system,
                                  temperature=temperature,
                                  response_format={"type": "json_object"})
        return schema.model_validate_json(raw)

    async def close(self) -> None:
        await self._client.aclose()

def create_llm_provider(*, api_key: str, model: str = "default", **kwargs):
    return MyLLMProvider(api_key=api_key, model=model, **kwargs)

Registering a Custom Provider

[providers.llm]
package = "my_llm_provider"          # Python module name (importable)
factory = "create_llm_provider"      # Factory function name
model = "my-model-v2"                # Passed as kwarg to factory
base_url = "https://my-api.com"      # Extra keys passed as **kwargs

The package value must be importable via importlib.import_module():

  • A Python file on your PYTHONPATH
  • An installed package (e.g., pip install my-llm-provider)
  • A built-in provider path (e.g., vne_cli.providers.llm.openai_provider)

Error Types

Providers should raise these errors from vne_cli.providers.errors when appropriate:

ErrorWhen to Raise
ProviderAuthErrorAPI key missing, invalid, or expired (HTTP 401/403)
ProviderRateLimitErrorRate limit exceeded (HTTP 429). Include retry_after if available.
ProviderResponseErrorUnexpected response format, server error (HTTP 5xx), or timeout
ProviderNotFoundErrorPackage not installed or factory function not found

VNE-CLI's retry logic uses exponential backoff with jitter: 3 retries, 1s base delay, 30s max delay, respects Retry-After header.

Configuration

VNE-CLI uses layered TOML configuration with this precedence (highest wins):

  1. CLI flags
  2. Environment variables (VNE_CLI_* prefix)
  3. Project config (./vne-cli.toml in your working directory)
  4. User config (~/.vne-cli/config.toml)
  5. Built-in defaults

Dependency Flow

Dependencies flow inward: cli -> commands -> {extraction, assets, assembly} -> {providers, flow, schemas} -> utils. No cycles. schemas/ and providers/ never import from extraction/, assets/, or assembly/.

Key Features

  • Three-stage pipeline: Extract, generate assets, assemble -- each stage independent and resumable
  • Pluggable providers: Use OpenAI, Anthropic, DALL-E, Stable Diffusion, or write your own
  • Smart text chunking: Respects chapter boundaries with context overlap across chunks
  • Character extraction: Automatic identification, deduplication, and expression variant detection
  • Branch detection: Explicit choice markers and implicit narrative branching cues
  • Cinematic direction: Automatic transitions, letterboxing, and timing from prose cues
  • Asset resume: Image generation tracks progress per-asset; resume after failures
  • Layered configuration: Global user config, per-project config, environment variables, and CLI flags

Flow Format

The .flow file format is what VoidNovelEngine uses to define scene logic. Each .flow file is a UTF-8 encoded JSON object containing nodes (operations) connected by links (execution and data flow).

Top-Level Structure

{
  "max_uid": 42,
  "node_pool": [ ... ],
  "link_pool": [ ... ],
  "is_open": true
}
FieldTypeDescription
max_uidintegerHighest ID assigned. Next entity gets max_uid + 1.
node_poolarrayArray of node objects.
link_poolarrayArray of link objects connecting pins between nodes.
is_openbooleanWhether this flow tab is open in the editor.

Control Flow

branchConditional branch (true/false)
loopRepeating execution
switch_sceneJump to another flow file

Presentation

Type IDDescription
delayPause for N seconds
wait_interactionWait for click/spacebar
switch_backgroundChange background image
add_foregroundAdd character/sprite
remove_foregroundRemove character/sprite
move_foregroundAnimate sprite movement
show_letterboxingShow cinematic bars
hide_letterboxingHide cinematic bars
show_subtitleDisplay subtitle text
show_dialog_boxShow dialogue box
hide_dialog_boxHide dialogue box
transition_fade_inFade in from black

Audio

Common Patterns

Dialogue Sequence

entry -> show_dialog_box("Elena", "Hello.") -> show_dialog_box("Marcus", "Hi there.")

Branching Choice

entry -> show_choice_button("Option A", "Option B", "", "", "")
           |                    |
           v                    v
     save_global          save_global
           |                    |
           +-----> merge_flow <-+
                      |
                      v
                  (continue)

Scene Transition

transition_fade_out(1.0) -> switch_scene("ch_001_sc_002")

Cinematic Narration

show_letterboxing(200, 1.5) -> show_subtitle("The dust settled...") -> hide_subtitle -> hide_letterboxing(1.5)

Link Direction (Important)

In the VNE link format, field names describe the link's perspective, not the pin's:

  • input_pin_id = the pin feeding data into the link = a source node's output pin
  • output_pin_id = the pin receiving data out of the link = a destination node's input pin

Data flows from input_pin_id to output_pin_id. The naming is the opposite of what you might expect.
